# React 前端重构计划

## 1. 项目现状分析

### 1.1 当前架构
*   **渲染模式**: 服务端渲染 (SSR)，使用 Google CTemplate 模板引擎。
*   **后端技术**: C++ (cpp-httplib) 直接返回 HTML 字符串。
*   **前端技术**: 原生 HTML/CSS/JS，依赖 jQuery (推测) 或原生 JS，缺乏模块化。
*   **路由管理**: 后端控制路由跳转，用户体验存在页面刷新等待。
*   **代码耦合**: 视图逻辑 (`oj_view.hpp`) 与业务逻辑紧密耦合，维护困难。

### 1.2 现有问题
*   **用户体验**: 每次跳转都需要重新加载页面，响应速度受网络影响大。
*   **开发效率**: 前后端代码混杂，前端修改需要重新编译后端服务。
*   **可维护性**: 缺乏组件化，样式和脚本分散，难以复用。
*   **扩展性**: 难以接入现代前端生态（如富文本编辑器、图表库）。

## 2. 技术选型方案

### 2.1 核心栈
*   **框架**: React 18 (利用并发特性和 Hooks)。
*   **构建工具**: Vite (极速开发体验)。
*   **语言**: TypeScript (强类型保障，减少运行时错误)。
*   **路由**: React Router v6 (标准单页应用路由)。
*   **状态管理**: Zustand (轻量级) 或 TanStack Query (服务端状态管理)。
*   **样式方案**: Tailwind CSS (原子化 CSS，快速开发) + shadcn/ui (高质量组件库)。
*   **HTTP 客户端**: Axios (拦截器处理统一认证和错误)。

### 2.2 目录结构规划
```
frontend/
├── src/
│   ├── assets/       # 静态资源
│   ├── components/   # 公共组件 (Button, Input, Navbar)
│   ├── features/     # 业务功能模块 (Auth, Problem, Contest)
│   ├── hooks/        # 自定义 Hooks
│   ├── layouts/      # 页面布局
│   ├── pages/        # 路由页面
│   ├── services/     # API 接口定义
│   ├── types/        # TypeScript 类型定义
│   ├── utils/        # 工具函数
│   ├── App.tsx       # 根组件
│   └── main.tsx      # 入口文件
├── public/           # 公共静态文件
├── index.html
├── package.json
├── tsconfig.json
└── vite.config.ts
```

## 3. 迁移策略

采用 **"绞杀者模式" (Strangler Fig Pattern)** 进行渐进式迁移，而非一次性重写。

1.  **独立开发**: 在项目根目录下新建 `frontend` 目录，独立开发 React 应用。
2.  **API 优先**: 确保后端 `/api/*` 接口功能完善，满足前端数据需求。
3.  **双轨运行**: 开发期间，旧版 SSR 页面保持不变，新版 React 应用通过开发服务器 (HMR) 预览。
4.  **逐步替换**: 
    *   **阶段一**: 替换静态展示页 (首页、关于页)。
    *   **阶段二**: 替换核心交互页 (登录/注册、题目列表)。
    *   **阶段三**: 替换复杂逻辑页 (题目详情、提交、竞赛)。
    *   **阶段四**: 替换管理后台。
5.  **最终切换**: 当所有功能迁移完毕，后端仅保留 API 接口，根路径 `/` 指向 React 构建产物 `index.html`。

## 4. 分阶段实施步骤

### 阶段 0: 基础设施搭建 (预计 3 天)
*   [ ] 初始化 Vite + React + TypeScript 项目。
*   [ ] 配置 ESLint, Prettier, Husky 代码规范。
*   [ ] 配置 Tailwind CSS 和基础组件库。
*   [ ] 封装 Axios 请求类 (处理 Cookie/Token, 统一错误提示)。
*   [ ] 配置 React Router 基础路由结构。

### 阶段 1: 公共组件与布局 (预计 5 天)
*   [ ] 实现全局 Layout (Navbar, Footer, Sidebar)。
*   [ ] 迁移 `navbar.html` 逻辑 (登录状态判断、头像显示)。
*   [ ] 实现基础 UI 组件 (Loading, Modal, Toast, Pagination)。
*   [ ] 对接 `/api/user` 接口，实现全局用户状态管理 (AuthContext)。

### 阶段 2: 核心功能迁移 (预计 10 天)
*   [ ] **首页**: 迁移 `index.html`，展示公告和热门题目。
*   [ ] **登录/注册**: 迁移 `login.html`，对接 `/api/login`, `/api/register`。
*   [ ] **题目列表**: 迁移 `all_questions.html`，实现分页、搜索、难度筛选。
*   [ ] **个人中心**: 迁移 `profile.html`，展示提交记录和个人信息。

### 阶段 3: 复杂业务迁移 (预计 14 天)
*   [ ] **题目详情**: 迁移 `one_question.html`。
    *   集成 Monaco Editor (代码编辑器)。
    *   实现 Markdown 渲染 (题目描述)。
    *   对接提交和评测接口 `/judge/:id`。
*   [ ] **讨论区**: 迁移 `discussion.html`，实现发帖、评论功能。
*   [ ] **训练/竞赛**: 迁移 `training` 和 `contest` 相关页面。

### 阶段 4: 游戏与后台集成 (预计 5 天)
*   [ ] **游戏**: 将 `games/` 目录下的静态游戏通过 `iframe` 或重构方式接入。
*   [ ] **管理后台**: 考虑暂时保留原 HTML 或简单重构为 React 路由的一部分。

### 阶段 5: 测试与部署 (预计 5 天)
*   [ ] 编写关键组件单元测试 (Vitest)。
*   [ ] 端到端测试 (Playwright) 覆盖核心流程。
*   [ ] 生产环境构建优化 (Gzip, CDN)。
*   [ ] 后端路由调整：将所有非 API 请求指向 `index.html` (解决 History 模式 404 问题)。

## 5. 风险评估与应对措施

| 风险点 | 影响等级 | 应对措施 |
| :--- | :--- | :--- |
| **SEO 影响** | 中 | 题目内容对 SEO 有要求，可考虑后续引入 Next.js 或预渲染 (Prerendering)。目前毕业设计场景可忽略。 |
| **首屏加载慢** | 低 | 使用代码分割 (Code Splitting)，路由懒加载，静态资源 CDN 加速。 |
| **API 不兼容** | 高 | 后端 C++ 接口可能与前端预期 JSON 格式不符。需后端配合调整，或前端增加适配层 (Adapter)。 |
| **样式冲突** | 中 | 使用 CSS Modules 或 Tailwind CSS 彻底隔离样式，避免全局污染。 |
| **开发进度滞后** | 中 | 严格执行敏捷开发，每日站会同步进度，优先保证核心路径 (Happy Path) 通畅。 |

## 8. 测试验收标准

1.  **功能完整性**: 所有旧版页面功能在 React 版中均可正常使用。
2.  **性能指标**: 首屏加载时间 (LCP) < 1.5s，交互响应时间 (INP) < 200ms。
3.  **兼容性**: Chrome, Firefox, Edge, Safari 最新版正常显示。
4.  **代码质量**: 无 TypeScript 报错，ESLint 检查通过，关键路径测试通过。

## 9. 回滚预案

1.  **代码层面**: 保持 `git` 分支独立，重构代码在 `feature/frontend-refactor` 分支进行。
2.  **部署层面**:
    *   保留原有 CTemplate 渲染逻辑作为备份。
    *   使用 Nginx配置灰度发布，通过 Cookie 或 Header 控制流量打入新/旧版本。
    *   若新版出现严重 Bug，立即切回旧版路由配置。
