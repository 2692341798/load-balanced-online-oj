# 第6章 系统测试

## 6.1 测试的目的

系统测试是软件开发生命周期中至关重要的环节，旨在全面验证基于多编译服务器负载均衡机制的在线评测系统的功能完整性、性能稳定性和安全性。本次测试的主要学术目标包括：

1. **功能正确性验证**：验证系统核心功能（如用户注册登录、题目列表获取、多语言代码提交、判题结果返回等）是否符合需求规格说明书的要求。
2. **高可用与容错性测试**：评估系统在异常情况下的恢复能力，验证负载均衡模块的心跳检测与调度机制的有效性。
3. **性能与并发承载评估**：测试系统在高并发场景下的吞吐量与响应延迟，寻找系统性能瓶颈；评估沙箱环境对代码运行时间、内存占用的限制能力，确保系统资源安全。
4. **用户体验一致性**：通过 UI 自动化测试，确保前端界面在不同场景下的渲染一致性。

## 6.2 测试方法

为保证测试结果的客观性与可重复性，本次测试采用自动化端到端测试方案，实现黑盒级别的业务场景验证。

1. **自动化测试框架**：采用 Playwright 作为核心测试框架。该框架支持多浏览器引擎，能够直接操作文档对象模型节点，并提供网络拦截等高级特性，适合现代前后端分离架构的集成测试。
2. **测试环境配置**：测试操作系统为 macOS 与 Linux，测试浏览器为 Chromium，目标环境为本地部署的完整前后端及数据库环境。
3. **测试数据准备**：通过测试脚本自动注入测试数据（如标准 C++、Java 代码，死循环代码，超大内存分配代码等），模拟真实用户的各种合法及非法操作。测试指标通过 JSON 报告生成器自动收集分析。

## 6.3 测试用例

本系统测试用例按照功能模块与测试场景进行分类，涵盖了系统的核心业务流。本次测试共覆盖8个核心业务场景，执行了28个测试用例，业务场景覆盖率达到100.00%。

### 6.3.1 功能与集成测试

本模块针对题目提交、沙箱隔离以及综合业务场景进行全面测试：

1. **正常通过**：输入标准算法代码、多组输入代码，预期系统返回“通过”。
2. **答案错误**：输入逻辑错误代码，预期系统返回“未通过”。
3. **时间超限**：输入死循环代码，验证沙箱对执行时间的限制。
4. **内存超限**：输入分配超大内存的代码，验证沙箱内存隔离机制。
5. **编译错误**：输入存在语法错误的代码，预期系统正确拦截并返回“编译错误”。
6. **多语言支持测试**：分别使用 C++、Java 等语言提交代码，验证系统多语言环境的兼容性。
7. **用户认证与状态同步**：验证登录、注册、鉴权及过期失效机制，同时验证用户提交后个人数据与界面渲染的正确性。

### 6.3.2 性能与并发测试

利用并发脚本模拟高频请求，验证后端负载均衡调度器在并发场景下的请求分发与服务吞吐能力。本次性能测试设定压测时段为30秒，重点监测系统的并发承载能力与响应延迟趋势。

在13并发度的标准压测场景下，系统在30秒内成功完成提交总数达到242次，95%请求（P95）响应时间为1913 ms，等价每秒查询率（QPS）为8.6，期间未出现任何服务端错误。

## 6.4 测试结论

本次自动化测试执行情况定量分析如下：执行用例总数28个，成功用例数28个，失败用例数0个，测试通过率为100.00%，整体测试流程耗时72.24秒，核心业务场景覆盖率为100.00%。

测试结果表明，系统各项功能均达到预期设计目标。沙箱隔离机制能够有效拦截超时与超内存的恶意代码，多语言编译模块运行稳定。在性能与并发方面，系统在13并发度下能够保持8.6 QPS的高效处理能力，且P95延迟控制在2秒内，表现出良好的可用性；然而，梯度压测客观暴露了系统在40并发度及以上的极限场景下存在处理瓶颈，请求出现超时。该瓶颈主要源于单节点评测机沙箱的资源分配与高并发执行开销，后续可通过增加编译服务器节点数量、优化负载均衡算法或引入消息队列进行削峰填谷等架构演进方式进行性能调优。总体而言，系统整体稳定可靠，完全满足在线评测系统的设计要求与上线标准。
