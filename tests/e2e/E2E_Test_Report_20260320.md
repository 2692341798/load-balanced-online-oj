### 3. 详细缺陷与失败原因分析

在本次运行的 28 个测试用例中，共发现 9 个失败用例，主要归结为以下 4 类核心缺陷：

#### 缺陷 1：登录页面 DOM 元素选择器失效 (严重)
- **影响范围**: `login.spec.ts` (所有5个用例)
- **错误日志**: `TimeoutError: locator.fill: Timeout 5000ms exceeded. Call log: waiting for locator('#login-username')`
- **问题分析**: 前端代码可能近期进行了重构，导致登录页面的用户名输入框 (`#login-username`)、密码输入框 (`#login-password`) 或提交按钮 (`#login-submit-btn`) 的 ID 发生了变化，测试脚本无法在规定的 5 秒内定位到该元素。
- **建议修复**: 开发人员需核对 `login.tsx`/`login.html` 中的表单元素 ID，或同步更新 `login.spec.ts` 中的定位器。

#### 缺陷 2：多语言选择器元素失效 (高)
- **影响范围**: `integration.spec.ts` (多语言支持场景)、`load.spec.ts` (负载测试)
- **错误日志**: `Expect "toBeVisible" locator('#language-select') failed. Error: element(s) not found`
- **问题分析**: 与缺陷 1 类似，题目详情页的语言下拉选择框（原选择器为 `#language-select`）未能在页面上找到，导致自动化脚本无法切换语言（如 C, C++, Python, Java）或注入对应代码。
- **建议修复**: 检查题目详情页的代码编辑器顶部工具栏，修复或同步 `#language-select` 元素的定位属性。

#### 缺陷 3：内存超限 (MLE) 判定与沙箱隔离异常 (高)
- **影响范围**: `judge.spec.ts` (4. Memory Limit Exceeded -> Case 1 & Case 3)
- **错误描述**: 测试用例提交了“超大 malloc 分配”和“超大静态数组”的恶意代码，预期系统返回“内存超限 (Memory Limit Exceeded)”，但实际并未正确匹配到该结果。
- **问题分析**: 
  1. 后端 C++ 评测机的沙箱 (Sandbox) 针对内存限制的底层调用（如 `setrlimit(RLIMIT_AS, ...)` 或 `cgroups`）可能未生效，导致进程未被操作系统主动 `SIGSEGV` 杀死。
  2. 或者返回了 `Runtime Error` 等其他状态码，而前端展示/测试脚本未将其正确识别为内存超限。
- **建议修复**: 审查后端 `compiler.hpp` 或评测核心逻辑中的内存限制分配参数，并确认判题进程捕获 `SIGSEGV` 信号后的状态映射逻辑。

#### 缺陷 4：视觉回归 (Visual Regression) 差异 (中)
- **影响范围**: `integration.spec.ts` (视觉回归测试)
- **错误描述**: 评测结果面板的实际截图与 Playwright 保存的基准快照 (Snapshot) 不匹配，差异率超出了允许的 `1%` 容差。
- **问题分析**: 系统最近可能修改了页面的 CSS 样式（例如深色主题、字体、边距或 Highlight.js 的渲染效果），导致像素级对比失败。
- **建议修复**: 如果近期确实调整了 UI，请在本地运行 `npx playwright test --update-snapshots` 以更新基准图片。

### 5. 总结与下一步计划
